# WindySail v0.3 — Identity, i18n, UX overhaul

Status: **approved brainstorm → ready for implementation planning**
Date: 2026-04-19
Branch: `v2-routing-ux-overhaul`

This spec bundles six sub-projects that together form the v0.3 release. Each
is independently implementable; ordering below reflects risk + dependency.

1. **F — Robustness fixes** (polar audit, swell-fetch watchdog)
2. **A — Backend identity + disclaimer** (Firebase)
3. **B — Internationalisation**
4. **E — UX polish pack** (From/To card, map markers, Done button, waypoint chart marks, footer version, right-click menu)

6. **C + D — Motorboat mode + Advanced routing settings**

Each sub-project gets its own implementation plan (via `superpowers:writing-plans`).
This spec is the shared source of truth.

## F — Robustness fixes

### F-1 Polar audit pass

Goal: verify the polar-speed pipeline is correct and unit-consistent.
Scope: code audit + inline fixes; no new tests or instrumentation.

Audit the following stack across `src/routing/Polar.ts`, `src/routing/IsochroneRouter.ts`, `src/adapters/WindProvider.ts`, `src/data/polarRegistry.ts`:

1. **Units.** Windy's `wind.speed` is metres/second. Polars are indexed in knots. Confirm every `getSpeed(polar, twa, tws)` call site passes **knots**. If any passes m/s, speeds roughly halve. Add a `knots(mps: number): number` helper in `src/routing/geo.ts` (or an appropriate utility module) and normalise at provider boundaries.
2. **TWA folding.** `Math.abs(((twa % 360) + 360) % 360)` on a signed input must produce the same angle as the unsigned form. Validate with edge cases: −10°, 190°, 370°, −370°.
3. **Axis ordering.** `speeds[twaLo][twsLo]` assumes `speeds[twaIndex][twsIndex]`. Confirm the parser in `polarRegistry.ts` produces that ordering for every bundled `.pol` file.
4. **Modifier stack.** The speed pipeline is `boatSpeed = getSpeed() → × (1 − swellPenalty) → motor override → + current vector → SOG`. Walk through a representative dev-mode route and log each stage once for a quick eyeball (remove logs after audit).

**Deliverable:** either inline fixes with a commit message describing what was wrong, or a commit `"polar audit: no defects found"` if the audit clears.

### F-2 Swell fetch watchdog

Goal: prevent indefinite hangs during swell/current sampling.
Root cause of current hang: each timestep awaits `waitForProductReady` + `waitForRedraw(150)` + `getLatLonInterpolator()` + `Promise.all(interpolate(...))`, none with a wall-clock bound. Only user-triggered AbortSignal interrupts.

**Change in `src/adapters/OceanDataProvider.ts`:**

- Wrap each timestep's async chain inside `_sampleSwellInner` and `_sampleCurrentInner` in a `Promise.race` against a per-step wall-clock timeout.
- On timeout, retry the step up to `STEP_RETRIES` times with `STEP_RETRY_BACKOFF_MS` backoff.
- On final failure: log a warning, treat the step as "no data" (increment `consecutiveEmpty`). The existing "2 consecutive empty → break" logic handles termination; `coverageEndTime` already flows downstream so the rest of the pipeline copes with partial data.
- AbortSignal handling unchanged.

**Tunables (exported constants at top of file):**
```ts
const STEP_TIMEOUT_MS = 15_000;
const STEP_RETRIES = 2;
const STEP_RETRY_BACKOFF_MS = 500;
```

## A — Backend identity + disclaimer

### A-1 Stack

Firebase + Firestore + HTTPS Cloud Functions. Region `australia-southeast1`.

Client **never** writes to Firestore directly. All mutations go through Cloud Functions. Firestore security rules deny all client reads and writes.

### A-2 Client identity

- `deviceId`: UUID v4, generated on first plugin open, stored in `localStorage.windysail-device-id`. **Primary key** in backend.
- `email`: `store.get('user')?.email ?? null`. Best-effort secondary field. Never the primary key.
- `pluginVersion`: imported from `package.json`.
- `usedLang`: `store.get('usedLang')`.
- `userAgent`: `navigator.userAgent` truncated to 200 chars.

If `localStorage` is unavailable (private browsing, quota): fall back to an in-memory UUID for the session. Do not block plugin functionality.

### A-3 Endpoints

All accept/return JSON. All writes are fire-and-forget from the client with a 5s timeout. **Never block plugin UX on backend response.** Failed POSTs are queued in `localStorage.windysail-pending-events` (max 50 entries, FIFO trimming) and replayed on next open.

| Route | Trigger | Payload |
|---|---|---|
| `POST /install` | First open on this device (no deviceId in localStorage) | `{deviceId, email?, pluginVersion, usedLang, userAgent}` |
| `POST /heartbeat` | Every subsequent open, debounced to max 1/day/device (see note) | `{deviceId, email?, pluginVersion, usedLang}` |

Heartbeat debounce: client stores `lastHeartbeatAt` (ISO string) in `localStorage.windysail-last-heartbeat`. `/heartbeat` POST is only fired on open if `(now - lastHeartbeatAt) >= 24h` or no record exists. Updated locally on successful POST; failed POSTs are queued per A-3 (they don't count as "sent" for the debounce clock).
| `POST /disclaimer-ack` | First press of Calculate Route with unacknowledged disclaimer version | `{deviceId, email?, pluginVersion, disclaimerVersion, acceptedAt}` |

CORS restricted to `https://www.windy.com` and `https://embed.windy.com`.

### A-4 Firestore schema

```
devices/{deviceId}
  email: string | null
  firstSeenAt: Timestamp
  lastSeenAt: Timestamp
  lastVersion: string
  lastLang: string
  installUserAgent: string
  disclaimerVersion: string | null
  disclaimerAcceptedAt: Timestamp | null

events/{autoId}
  deviceId: string
  type: 'install' | 'heartbeat' | 'disclaimer-ack' | 'coffee'
  payload: map
  receivedAt: Timestamp
  ipHash: string   // sha256 of (IP + server-side salt), server-derived
```

### A-5 Repo layout

```
/firebase/
  functions/
    src/
      index.ts          # Express router on single HTTPS function
      install.ts
      heartbeat.ts
      disclaimer.ts
      schemas.ts        # zod request schemas
    package.json
    tsconfig.json
  firestore.rules        # deny all client reads/writes
  firebase.json
  .firebaserc
```

### A-6 Disclaimer UX

- Disclaimer text lives in i18n (`disclaimer.body` key) — translatable.
- `DISCLAIMER_VERSION` constant in `src/disclaimer.ts` (e.g. `"2026-04"`).
- On **Calculate Route** press: check `localStorage.windysail-disclaimer-ack` matches current version. If not, show modal and block Calculate.
- Modal content: title, body, checkbox "I understand this is not for navigation", **Accept & Calculate** button (disabled until checkbox ticked).
- On accept: store locally (`{version, acceptedAt}`), POST `/disclaimer-ack`, then proceed with Calculate.
- The modal does **not** block plugin browsing — users can still look around; only Calculate is gated.

### A-7 Future hooks

Not built now but the schema accommodates:

- `POST /coffee` — records `{deviceId, clickedAt}` for Buy Me a Coffee button analytics.
- Admin view as a Next.js / static page on Firebase Hosting, reading with service-account creds.

## B — Internationalisation

### B-1 Strategy

- Extract every user-visible string into typed i18n files.
- Ship **English only** at launch; structure supports drop-in additional locales.
- Brodie will generate other locales via LLM using the English source file as input.
- Planned target locales (files added empty or LLM-seeded over time): `fr`, `es`, `it`, `de`, `pt`, `el`, `nl`, `hr`, `pl`.

### B-2 File layout

```
src/i18n/
  en.ts              # source of truth
  fr.ts / es.ts / ... # created post-launch
  types.ts           # type Translations = typeof import('./en').en
  index.ts           # t(), locale store, setLocale, available()
```

`fr.ts`, `es.ts`, etc., use `satisfies Translations` so TypeScript flags missing keys.

### B-3 English source shape

```ts

    acceptButton: 'Accept & Calculate',
  },

  errors: {
    swellUnavailable: 'Swell data unavailable — continuing without it.',
    windUnavailable: 'Wind data unavailable for {model}.',

  footer: { versionPrefix: 'v' },

```

Nested keys grouped by UI section. Placeholders use `{name}` syntax.

### B-4 Helper API

```ts
import { t, locale } from '../i18n';

t('routing.calculateButton');
t('routing.startInstruction', { what: t('routing.startWhat') });
```

- `t(key: string, params?: Record<string, string | number>): string`
- `locale` — Svelte readable store reflecting `store.get('usedLang')`; emits on Windy language changes.
- **Fallback chain:** requested locale → `en` → raw key (visible bug indicator).
- Unknown or missing locale → console warn in dev, fall back to `en`.
- **No string concatenation of translated fragments.** Always use placeholders.

### B-5 Scope of conversion

Every hard-coded user-visible string in `src/ui/*.svelte`, `src/plugin.svelte`, error messages in adapters, and console warnings surfaced to the user must be converted to `t('…')`. Stack traces and dev-only logs stay in English.

## G — GPX export

### G-1 Trigger

**Export GPX** button in the results panel area (same row as Save Route / model compare controls). Label i18n key: `routing.exportGpx`.

### G-2 Content

Export the **currently selected route** only (the model the user has highlighted). A single `<rte>` container holding the Start / waypoints / Finish as `<rtept>` elements. No timing, no wind data, no separate `<wpt>` markers.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<gpx version="1.1" creator="Windy Sail Router v{version}"
     xmlns="http://www.topografix.com/GPX/1/1">
  <metadata>
    <name>{startName} to {endName}</name>
    <time>{isoTimestamp}</time>
  </metadata>
  <rte>
    <name>Sail Router plan</name>
    <rtept lat="{lat}" lon="{lon}"><name>Start — {startName}</name></rtept>
    <rtept lat="{lat}" lon="{lon}"><name>Waypoint N</name></rtept>
    <rtept lat="{lat}" lon="{lon}"><name>Finish — {endName}</name></rtept>
  </rte>
</gpx>
```

### G-3 Mechanics

- New file `src/export/gpxExport.ts`:
  - `buildGpx(args: { startName, endName, start, end, waypoints, version }): string`
  - `escapeXml(s: string): string` for safe text interpolation.
  - No XML library dependency — template literals.
- Download trigger: `Blob` → `URL.createObjectURL` → `<a download="…"` click → `URL.revokeObjectURL`.
- Filename: `sail-router-{slugify(startName)}-{slugify(endName)}-{YYYY-MM-DD}.gpx`.
- Unit test: golden-file comparison against `tests/fixtures/route-3wpt.gpx` for a fixed three-waypoint input.

```ts
motorboatHeavyKt: number         // default 5
motorboatSwellThresholdM: number // default 2.5
```

UI: three numeric inputs in the boat card when `motorboatMode === true`.

### C-3 Routing algorithm change

In `src/routing/IsochroneRouter.ts`, when `motorboatMode`:

```ts
boatSpeed = (swellHeight > motorboatSwellThresholdM)
  ? motorboatHeavyKt
  : motorboatCruiseKt;
```

- **Skipped:** polar `getSpeed()`, swell speed reduction (already baked into the threshold curve), motor-override block (the whole route is motored).
- **Still applied:** current vectorisation into SOG.
- **Not applied:** tack / gybe penalties (C + D interaction — motor mode is direction-agnostic).

## D — Advanced routing settings

### D-1 UI

New collapsible **Advanced** section in `SettingsModal`. Closed by default. Summary line shown collapsed:
*"Tack / gybe penalties, wind limits, night sailing, reefing"*.

### D-2 Settings additions

```ts
advanced: {
  tackPenaltyS: number            // default 15
  gybePenaltyS: number            // default 20
  motorAboveTws: number | null    // default null (disabled)
  motorBelowTws: number | null    // default 4
  nightSpeedFactor: number        // default 1.0 (no reduction); range 0.5 – 1.0
  reefAboveTws: number | null     // default null (disabled)
  reefFactor: number              // default 0.85
}
```

### D-3 Algorithm wiring

Implemented in `IsochroneRouter.ts`. **Skipped entirely in motorboat mode** (D-values irrelevant when C is active).

**Tack / gybe detection.** In the isochrone expansion loop:

- Compare the new leg's TWA sign to the previous leg's TWA sign (on the same parent node).
- Sign crossing through 0° = **tack** → `legDuration += tackPenaltyS`.
- Sign crossing through ±180° = **gybe** → `legDuration += gybePenaltyS`.
- No penalty when staying on the same tack.

**TWS limits.** After polar lookup, before `comfortWeight`:

- If `tws > motorAboveTws` or `tws < motorBelowTws`, and `motorOptions.enabled`, force `boatSpeed = motorOptions.speed; isMotoring = true`.

**Reef factor.** If `tws > reefAboveTws && !isMotoring` → `boatSpeed *= reefFactor`. Proxy for heel-angle / reefing (polars don't carry heel data). Stacks with subsequent motor evaluation.

**Night reduction.** Compute whether the leg's midpoint time + lat/lon is between civil dusk and civil dawn using an inline sun-altitude formula (or a `@windy/` helper if one exists — check during implementation). If so: `boatSpeed *= nightSpeedFactor`.

### D-4 Order of operations per leg

```
1. getSpeed(polar, twa, tws)                 — raw polar
2. reef factor (if tws > reefAboveTws)
3. night factor (if sun below horizon at leg midpoint)
4. swell reduction (existing)
5. motor override (existing motor + D-3 TWS limits)
6. add current vector → SOG
7. compute leg duration; add tack/gybe penalty if transition detected
```

### D-5 UI details

- Numeric inputs with explanatory tooltips.
- **TWS limits** row: two fields (above / below) plus a master toggle that disables both when off.
- **Night mode**: slider 50% – 100% with current value label.
- **Reefing**: `reefAboveTws` numeric + `reefFactor` slider 0.5 – 1.0 (labelled as percentage reduction).

## Out of scope for v0.3

- **Buy Me a Coffee button** — schema supports it (A-7), UI is deferred.
- **Admin dashboard** — can be bolted on later as a Firebase Hosting Next.js page.
- **Uninstall tracking beyond heartbeat staleness** — unobservable from the plugin; server derives from lack of heartbeats.
- **Heel-angle routing from real polar heel data** — no standard `.pol` format carries heel; `reefAboveTws` is the v0.3 proxy.
- **Chart splitting by leg** — rejected in favour of overlay annotations (E-4).
- **Email-login gate** — rejected; `deviceId` is the primary key so anonymous Windy users still work.
- **Bundled translations at launch** — ship English only; locales come in post-launch.
